# Node Monolith 3-Tier Application

## Overview

This is a Node.js + React-based monolithic user management web application serving as the **source codebase** for two downstream DevOps projects:

- **[DevSecOps Pipelines](https://github.com/ibtisam-iq/devsecops-pipelines)** ‚Äî CI/CD pipelines that build, scan, and package this application into a secure, deployable artifact using Jenkins, GitHub Actions, Docker, SonarQube, and Trivy.
- **[Platform Engineering Systems](https://github.com/ibtisam-iq/platform-engineering-systems)** ‚Äî Deployment workflows that run this artifact across Docker Compose, AWS EC2, EKS (Kubernetes), Terraform, and GitOps-based delivery.

> I did not build this application from scratch. As a DevOps Engineer, my focus is on everything that happens **around the code** ‚Äî building, securing, packaging, and operating it in production-like environments.

> The files I added to this repository are: `Dockerfile.client`, `Dockerfile.server`, `compose.yml`, `.dockerignore`, `.gitignore`, and `.env.example`. Everything else under `client/` and `server/` belongs to the original developer ‚Äî with the exception of the architectural refactoring documented in Step 0 below.

Three-tier architecture: Presentation (React SPA served by Nginx ‚Äî Tier 1) ‚Üí Business Logic (Express + MVC ‚Äî Tier 2) ‚Üí Data (MySQL ‚Äî Tier 3).

> **Key distinction from the 2-tier project:** Express does **not** serve static files here. Nginx is the sole entry point (port 80) ‚Äî it delivers the compiled React build directly and proxies all `/api/` requests to Express on port 5000. The two services run in separate containers and never share a process.

Before doing any DevOps work, I audited and modernized the inherited codebase ‚Äî refactoring the backend from a flat 2-tier structure into a proper MVC architecture, separating routes, controllers, models, and a connection pool into dedicated files, and modernizing all dependencies.

> I used **AI-assisted analysis (Perplexity Pro)** for this step. Full audit notes, dependency changes, and migration details: [`docs/codebase-modernization.md`](docs/codebase-modernization.md)

As a baseline, this project is a direct evolution of **[node-monolith-2tier-app](https://github.com/ibtisam-iq/node-monolith-2tier-app)** ‚Äî the same application refactored from a single flat `server.js` into a clean three-tier MVC structure with Nginx as the presentation layer.

### Step 1 ‚Äî Environment Standardization

The original codebase had hardcoded database credentials directly in `config/db.js` and `server.js`. I refactored the entire application to read all config from environment variables, making it portable across all environments.

```bash
# Copy the template and fill in real values
cp .env.example .env
```

Key variables set in `.env`:

```env
MYSQL_ROOT_PASSWORD=your_root_password
MYSQL_DATABASE=test_db
MYSQL_USER=your_username
MYSQL_PASSWORD=your_password
DB_HOST=localhost
PORT=5000
```

> **Note:** For local bare-metal runs, `DB_HOST=localhost` is correct ‚Äî MySQL is running directly on the same machine. When running via Docker Compose, `DB_HOST` is overridden to `mysql` (the database service name defined in `compose.yml`) via the `environment:` block. Docker resolves that service name as a hostname on the internal container network, so `localhost` will not work there.

### Step 2 ‚Äî Local Build & Bare-Metal Validation

Before writing any Docker config, I validated the full application lifecycle locally ‚Äî native MySQL, native Node.js, and Nginx. This confirmed the build was clean and the app connected to the database correctly before introducing any containerization layer.

#### 1. Install and Configure MySQL

```bash
sudo apt update && sudo apt install -y mysql-server
sudo systemctl start mysql && sudo systemctl enable mysql
sudo mysql -u root -p
```

```sql
CREATE DATABASE test_db;
CREATE USER 'your_username'@'localhost' IDENTIFIED BY 'your_password';
GRANT ALL PRIVILEGES ON test_db.* TO 'your_username'@'localhost';
FLUSH PRIVILEGES;
EXIT;
```

```bash
# Bootstrap the schema
mysql -u your_username -pyour_password test_db < database/init.sql

# Verify
mysql -u your_username -pyour_password -e "USE test_db; SHOW TABLES;"
```

#### 2. Install Dependencies and Start the Backend

```bash
cd server && npm install
node server.js
```

The server printed:

```
Server running at http://localhost:5000
Database pool connected successfully.
```

#### 3. Build the Frontend

```bash
cd ../client && npm install
npm run build
```

This created `client/public/` with `index.html`, `bundle.js`, and `style.css` ‚Äî all generated by Webpack.

> The frontend must be built **before** starting Nginx. Nginx serves the compiled static files from `client/public/`. If `public/` does not exist, Nginx will return 404 for all page requests.

#### 4. Install and Configure Nginx

Nginx acts as the reverse proxy that unifies both tiers under a single port (80). It serves static files from `client/public/` and proxies all `/api/` requests to the Express server on port 5000.

```bash
sudo apt install -y nginx

# Deploy the project's Nginx config
sudo cp nginx/default.conf /etc/nginx/conf.d/default.conf

# Remove the default Nginx site (it also listens on port 80 and takes priority)
sudo rm -f /etc/nginx/sites-enabled/default

# Grant Nginx (www-data) access to the home directory and public folder
sudo chmod o+x ~
sudo chmod -R o+r ~/node-monolith-3tier-app/client/public

# Validate config syntax and restart
sudo nginx -t && sudo systemctl restart nginx
```

> **Why `chmod o+x ~`?** Nginx runs as `www-data`. To serve files from a path inside your home directory, it must be able to traverse each directory in that path. The home directory is typically `700` (owner-only), which blocks `www-data` from entering it. `chmod o+x ~` adds traverse permission for others without exposing home directory contents.

> **Why remove `sites-enabled/default`?** Nginx on Ubuntu ships with a default site that also binds to port 80. It silently takes priority over configs in `conf.d/`. Without removing it, your custom config is loaded but never used.

#### 5. Verify End to End

```bash
# Confirm the backend API is up (direct, bypasses Nginx)
curl http://localhost:5000/api/test

# Confirm the API is reachable through Nginx
curl http://localhost/api/test

# Confirm Nginx is serving the frontend
curl http://localhost
```

App runs at: `http://localhost`

### Step 3 ‚Äî Containerization (Docker)

With the application validated on bare metal, I wrote `Dockerfile.client`, `Dockerfile.server`, and `compose.yml` from scratch. I read `client/package.json`, `client/webpack.config.js`, `server/package.json`, `server/app.js`, and `.env.example` before writing a single line ‚Äî to understand exactly what each image needed: build stages, output paths, exposed ports, health endpoints, and environment variable strategy.

**Key decisions I made and documented:**

- **Two separate Dockerfiles** ‚Äî because Express does not serve static files, the frontend (Nginx) and backend (Node.js) must be independent images. This is the defining difference from the 2-tier project, which uses a single Dockerfile.
- **Multi-stage `Dockerfile.client`** ‚Äî Stage 1 builds the React app with Node.js + Webpack; Stage 2 copies only `client/public/` into `nginx:alpine`. Final image is ~25MB with zero Node.js in it.
- **Multi-stage `Dockerfile.server`** ‚Äî Stage 1 installs production-only dependencies (`npm install --omit=dev`); Stage 2 assembles the lean runtime image.
- **`nginx/docker.conf` instead of `nginx/default.conf`** ‚Äî inside Docker, `localhost` in `proxy_pass` would point to the Nginx container itself, not Express. `docker.conf` uses the Compose service name `server` as the hostname, which Docker's internal DNS resolves correctly.
- **`expose:` vs `ports:`** ‚Äî only the Nginx service maps a host port (`80:80`). Express uses `expose: "5000"` ‚Äî reachable within the Docker network only, never directly from the host.
- **Non-root user** in both server and client images for CIS/Trivy compliance.
- **Exec form `ENTRYPOINT`** on the server so Node.js runs as PID 1 and receives `SIGTERM` directly for graceful shutdown.
- **Healthcheck endpoints tuned per service** ‚Äî Nginx uses `GET /` (port 80); Express uses `GET /api/test` (port 5000, the only route that returns 200).

The full rationale for every line in all three files is in [`docs/docker-setup.md`](docs/docker-setup.md). The architectural reasoning behind why two Dockerfiles are required is in [`docs/understand-architecture.md`](docs/understand-architecture.md).

#### Validating with Docker Compose

After writing the files, I validated them end-to-end using Docker Compose ‚Äî spinning up all three services (Nginx, Express, MySQL) as containers on a shared internal network, with no local installations needed.

```bash
cp .env.example .env
# Fill in: MYSQL_ROOT_PASSWORD, MYSQL_DATABASE, MYSQL_USER, MYSQL_PASSWORD, PORT

docker compose up --build
```

> `--build` forces Docker to rebuild both images from their Dockerfiles. Omit it on subsequent runs if the source code has not changed.

**What happens in sequence:**
1. Docker builds `node-monolith-3tier-app-client` (Nginx image) using `Dockerfile.client`
2. Docker builds `node-monolith-3tier-app-server` (Express image) using `Dockerfile.server`
3. The `mysql` container (MySQL 8.4 LTS) starts and runs its healthcheck (`mysqladmin ping -h localhost`)
4. The `server` container waits for the `mysql` healthcheck to pass (`condition: service_healthy`)
5. Express connects to MySQL using the service name `mysql` as the hostname (overrides `localhost` from `.env`)
6. The `nginx` container starts and begins serving the React build and proxying `/api/` to `server:5000`
7. The app becomes available at `http://localhost`

**Verify the app is running:**

```bash
# Through Nginx ‚Äî the correct production path
curl http://localhost
curl http://localhost/api/users
```

**Stop and clean up:**

```bash
# Stop containers but keep the database volume
docker compose down

# Stop containers AND delete the database volume (full reset)
docker compose down -v
```

## Key Idea

> Code = Input. Pipelines secure it. Infrastructure runs it.

| Repository | Role |
|---|---|
| **This repo** | Application source code ‚Äî the single input to everything below |
| **[DevSecOps Pipelines](https://github.com/ibtisam-iq/devsecops-pipelines)** | CI/CD ‚Äî builds, scans, and packages the code into a deployable artifact |
| **[Platform Engineering Systems](https://github.com/ibtisam-iq/platform-engineering-systems)** | Platform ‚Äî deploys, operates, and scales the artifact across multiple targets |

This separation is intentional: one repo per concern. The source code stays clean, the pipeline logic stays auditable, and the deployment configs stay independently versioned.
